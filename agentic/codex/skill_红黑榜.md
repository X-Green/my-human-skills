# Agent Skills 团队选型红黑榜

> 评估时间：2026-07-14。本文评估的是团队适配性，而不是项目热度。上游维护状态和功能描述是事实判断；推荐等级、成本和风险是团队选型建议。

## 快速结论

| 对象 | 分组 | 核心价值 | 最适合 | 主要成本 | 采用建议 |
| --- | --- | --- | --- | --- | --- |
| `grill-me` | 红榜 | 用连续追问暴露方案中的假设和缺口 | 需求模糊、方案尚未定型 | 需要产品或技术负责人持续回答 | 作为重要变更的前置访谈工具 |
| `grill-with-docs` | 红榜 | 在追问过程中同步沉淀领域词汇和架构决策 | 长期维护的业务代码库 | 需要维护 `CONTEXT.md` 和 ADR | 在已有文档责任人的团队中优先试点 |
| `domain-modeling` | 红榜 | 建立统一领域语言，并校验文档、代码与口头认知 | 复杂业务、多团队协作 | 需要领域专家参与和持续治理 | 作为独立能力按需使用 |
| `superpowers` | 黑榜 | 为自主规划能力较弱的模型补齐端到端工程流程 | 旧模型或缺少基本规划约束的代理环境 | 小任务会被放大成大量流程任务，显著消耗时间和 token | GPT-5.6 及后续模型不建议整套使用；最多按缺口抽取单项 skill |

## 评价口径

- **红榜**：问题边界清楚、产物明确，能够较容易地嵌入团队现有流程。推荐先在真实任务中试点，再沉淀为团队惯例。
- **观察榜**：能力有价值，但流程侵入性、使用成本或与现有制度的重叠较高。需要限定范围验证，不建议默认全量启用。当前没有项目处于观察榜。
- **黑榜**：收益明显低于成本，或存在行为与当前模型能力不匹配、任务被过度放大、与团队关键制度冲突等问题。`superpowers` 因不适配 GPT-5.6 及后续模型而列入黑榜。

主要评价维度包括：问题匹配度、接入成本、流程侵入性、产物可维护性，以及与团队已有研发规范的重叠程度。不能仅凭版本年龄或个人使用习惯判断一个 skill “过时”。

## 红榜：推荐采用

### `grill-me` 与 `grill-with-docs`

**定位与差异**

`grill-me` 是一个轻量入口：它启动连续追问，通过一次只解决一个关键问题的方式，帮助团队在实现前澄清目标、约束、分支选择和隐含假设。它适合非代码问题，也适合尚未进入正式文档阶段的技术方案讨论。

`grill-with-docs` 在同一套追问流程上叠加 `domain-modeling`，会在讨论过程中维护领域词汇，并在满足条件时沉淀架构决策记录（ADR）。前者侧重“把问题问清楚”，后者还负责“把形成的共识留下来”。

**适用场景**

- 需求只有目标描述，验收标准、边界条件或优先级仍不明确。
- 技术方案存在多个分支，需要在编码前暴露关键取舍。
- 变更涉及产品、研发和领域专家，团队成员对术语或规则理解不一致。
- 代码库需要长期维护，希望把讨论结果同步沉淀为可复用上下文。
- PRD、技术方案或实施计划即将定稿，需要做一次结构化压力测试。

**不适用场景**

- 任务已经有明确规格、验收标准和实现路径，继续追问只会重复已有信息。
- 紧急且低风险的机械修改，例如确定范围内的文案或配置值调整。
- 没有能做最终决策的参与者；代理可以暴露问题，但不能代替业务负责人拍板。
- 团队没有维护 `CONTEXT.md` 和 ADR 的责任人，却准备直接使用 `grill-with-docs`。

**使用成本**

- 访谈需要关键人员持续参与，前期沟通时间会增加。
- `grill-with-docs` 会产生长期文档维护成本；代码或业务变化后，旧术语和旧决策也需要更新。
- 追问质量依赖代理是否充分读取仓库，以及参与者能否给出明确答案。

**风险与注意事项**

- 追问如果没有停止条件，容易把低风险任务过度设计。开始前应约定目标产物，例如“形成可评审方案”或“补齐验收标准”。
- 文档不能替代验证。凡是能从代码、测试或运行结果确认的事实，应先核查仓库，而不是让人凭记忆回答。
- 使用 `grill-with-docs` 时，应明确只做澄清与文档更新，还是允许讨论结束后继续实现，避免代理未经确认进入编码。
- ADR 应记录难以逆转、缺少背景会令人困惑且确实存在权衡的决定，不应把每次讨论都固化成 ADR。

**替代方案**

- 使用团队现有的需求评审清单、RFC 模板或架构评审会议。
- 对轻量任务，由负责人直接要求代理“一次只问一个最关键的问题”，不安装完整 skill。
- 已有成熟领域文档体系时，可保留现有模板，只借用连续追问的方法。

**团队采用建议**

- 默认先试点 `grill-me`，验证它能否减少返工和评审中的关键遗漏。
- 只有当代码库确实需要统一领域语言，并且文档有人维护时，再升级到 `grill-with-docs`。
- 把停止条件、允许写入的文档位置和实现权限写入团队使用约定。

### `domain-modeling`

**定位**

`domain-modeling` 不是被动读取词汇表，而是在设计过程中主动维护领域模型：发现含糊或冲突的术语，用具体场景验证概念边界，对照代码识别认知偏差，并在结论形成时更新 `CONTEXT.md`。

它要求 `CONTEXT.md` 只承载领域词汇和含义，不混入实现细节、临时笔记或完整需求。对于大型系统，可以通过 `CONTEXT-MAP.md` 划分多个限界上下文。ADR 则按需创建，记录难以逆转、缺少背景会令人困惑且存在真实权衡的架构决定。

**适用场景**

- 业务规则复杂，同一个词在产品、研发、运营之间含义不同。
- 多个团队或服务共同处理相近概念，需要明确边界和所有权。
- 代码中出现命名漂移，同一概念存在多个名称，或同一名称指代多个概念。
- 新成员和代理需要频繁重新理解项目术语，沟通成本长期重复发生。
- 架构讨论需要同时核对领域规则、现有代码和历史决策。

**不适用场景**

- 领域非常简单，通用术语已足以准确表达业务。
- 一次性脚本、实验性原型或即将废弃的模块，文档的长期收益有限。
- 团队只是想记录接口、部署或数据库细节；这些内容应进入对应技术文档，而不是 `CONTEXT.md`。
- 缺少领域专家参与，代理只能根据代码猜测业务含义。

**使用成本**

- 需要领域专家、技术负责人和文档维护者共同校准术语。
- 初次梳理会暴露历史不一致，可能带来命名迁移和文档整理工作。
- 领域模型会随业务演进，需要在重要变更中持续更新，而不是一次性生成后搁置。

**风险与注意事项**

- 代理可能把实现结构误当成业务事实，应通过真实业务场景和负责人确认。
- 过度追求统一词汇可能掩盖不同限界上下文中的合理差异。
- `CONTEXT.md` 如果混入大量技术细节，会失去作为精炼词汇表的价值。
- ADR 数量过多会增加检索和维护成本，应坚持按条件、按需创建。

**替代方案**

- 使用现有的领域词典、事件风暴产物、C4 模型说明或团队 Wiki。
- 在较小项目中，用一份受评审的 glossary 代替完整的多上下文结构。
- 如果团队已经有稳定的领域建模流程，只需让代理读取并遵守现有文档。

**团队采用建议**

- 先选择术语冲突最明显的一个业务域试点，不要一开始覆盖整个组织。
- 明确 `CONTEXT.md`、ADR 和技术文档各自的内容边界与负责人。
- 在领域规则或关键命名变更的评审清单中加入文档同步检查。

## 黑榜：不建议整套采用

### `superpowers`

**复评结论**

原“过时”如果指“上游停止维护”并不准确：核查时，上游仍在持续发布版本，并为 Codex、Claude Code、Cursor 等多种代理环境提供安装方式。

但维护活跃不等于适合当前模型。本团队的主要否决原因是：`superpowers` 会把一个小任务展开为需求澄清、设计文档、实施计划、工作树、多个子任务、TDD、分阶段评审和分支收尾等大量流程任务。对 GPT-5.6 及后续已经具备较强自主规划和验证能力的模型，这些强制步骤大量重复模型原生能力，显著增加完成时间和 token 消耗，实际收益不足以覆盖成本。

因此，`superpowers` 在本指南中进入黑榜，含义是“不适合 GPT-5.6 及后续模型整套使用”，而不是“项目无人维护”或“所有单项 skill 都没有价值”。

**适用场景**

- 使用 GPT-5.5 及更早、或自主规划能力明显不足的模型，需要外部流程补齐基本工程纪律。
- 只抽取一个边界明确的 skill，例如系统化调试或完成前验证，并关闭整套自动触发流程。
- 团队正在做受控对比实验，明确把额外时间和 token 作为评估成本，而不是直接用于日常开发。

**不适用场景**

- GPT-5.6 及后续模型。模型已经能按任务复杂度自主规划，固定的前置设计和计划流程容易形成重复劳动。
- 小型修改、文档编辑、配置调整和局部修复。这些任务最容易被放大成远超实现本身的流程工作。
- 对交付时延、token 预算或人工等待敏感的任务。
- 团队已有 RFC、测试、评审和发布制度，整套引入会同时重复模型能力与团队制度。
- 需要代理在明确授权后连续执行，而不是每个阶段都停下来创建产物、请求确认或切换工作流。

**使用成本**

- **任务数量膨胀**：一个原本可以直接完成的小任务，会派生出规格、计划、检查清单、多个执行步骤和收尾任务。
- **串行等待增加**：设计批准、规格复核、执行方式选择和分支处理等门禁会反复中断主任务。
- **token 消耗增加**：代理需要反复读取 skill、复述计划、维护中间文档并进行多轮状态汇报。
- **工具链开销增加**：worktree、分支、子代理和多阶段评审带来额外创建、同步与清理成本。
- **重复治理**：其规则可能同时复制 GPT-5.6 之后模型的原生能力和团队已有的研发规范。

**风险与注意事项**

- 强制触发和硬门禁可能压过用户对任务规模的判断，让代理为了遵守流程而制造并不存在的复杂度。
- 大量中间产物会稀释上下文中的真实需求，使简单任务更容易偏离目标。
- 执行时间和 token 的增长并不自动带来更高质量；许多步骤只是在重复规划与确认。
- 在模型能力升级后继续沿用为旧模型设计的完整流程，会让工作流成为性能瓶颈。
- 上游版本更新可能继续改变默认行为；即使只抽取单项 skill，也应固定并复评实际使用版本。

**替代方案**

- 直接使用 GPT-5.6 及后续模型的原生规划与执行能力，只在高风险节点增加明确检查。
- 使用 `grill-me` 处理真正模糊的需求，其余步骤接入团队已有的任务拆分、评审和发布流程。
- 将必要约束写入简短的仓库级指令，并用 CI、分支保护和代码审查保障不可妥协的规则。
- 确有缺口时，只选用 `systematic-debugging` 或 `verification-before-completion` 等单项能力，不加载完整方法论。

**团队采用建议**

- GPT-5.6 及后续模型不安装或停用整套 `superpowers` 自动触发流程。
- 盘点其中确有独立价值的 skill，按实际缺口建立白名单，不让单项使用重新串成完整流程。
- 用同一组小型、中型任务对比原生模型与候选 skill，记录总耗时、token、人工中断次数和结果质量。
- 只有单项 skill 在可重复测试中带来明确净收益时，才纳入团队基线。

## 团队落地建议

1. **从问题出发**：先列出团队最常见的代理失误，例如需求误解、术语混乱、跳过测试或交付未经验证，再选择对应 skill。
2. **单项试点**：优先试用边界清楚的 `grill-me` 或 `domain-modeling`，避免同时更换需求、开发和交付全流程。
3. **记录效果与摩擦**：在复盘中记录遗漏是否减少、产生了哪些可维护产物，以及新增了多少人工确认和流程冲突。不要只收集主观“好用”评价。
4. **形成团队基线**：明确触发条件、允许写入的位置、停止条件、人工审批点和产物负责人，并写入仓库级规范。
5. **谨慎组合**：需要长期领域知识时组合 `grill-with-docs`；GPT-5.6 及后续模型不要引入整套 `superpowers`，只有出现具体缺口时才评估单项 skill。
6. **周期性复评**：上游版本、宿主代理能力和团队流程都会变化。升级前检查行为变化，升级后用代表性任务重新验证。

Skill 应补充团队现有的评审、测试和发布制度，而不是重复、绕过或暗中替换这些制度。最终选型标准不是“流程看起来完整”，而是它能否以可接受的成本降低团队已经确认的风险。
